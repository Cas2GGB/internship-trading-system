# 项目代码执行逻辑与文件作用详解

本文档详细解析了 “交易系统快速内存快照方案原型” 的代码结构、执行逻辑及各文件的具体职责。

## 1. 目录概览

```
├── bin/                 # 编译产出目录 (executable)
├── build/               # CMake 构建中间文件
├── conf/                # **[新增]** 配置文件目录 (matching_engine.conf)
├── doc/                 # 项目文档
├── include/             # 头文件目录
│   ├── common/          # 通用基础库 (Logger, Order, Types, ObjectPool)
│   └── engine/          # 撮合引擎核心 (OrderBook, Engine, SkipList)
├── src/                 # 源代码目录
│   ├── main.cpp         # 程序入口
│   └── engine/          # 引擎实现 (.cpp)
└── test/                # 测试数据与脚本
```

## 2. 核心执行流程

### Step 1: 启动与初始化 (`src/main.cpp`)

程序从 `main` 函数开始执行。

1.  **实例化引擎**: 创建 `Engine` 类实例。
2.  **加载配置**: 调用 `engine.init(configPath)`，读取 `conf/matching_engine.conf`。
    *   **参数解析**: 解析 `INIT_STOCKS`（预加载股票ID列表）和 `SNAPSHOT_INTERVAL`（自动快照间隔）。
    *   **预热**: 根据配置列表，预先创建对应股票的 `OrderBook` 实例。
3.  **脚本加载 (可选)**: 如果命令行参数包含文件路径（如 `./matching_engine test_scenario.txt`），则调用 `engine.loadScript()` 预先执行一批指令。
4.  **进入事件循环**: 调用 `engine.run()`，开始监听标准输入 (`stdin`) 的指令。

### Step 2: 引擎指令处理 (`src/engine/Engine.cpp`)

`Engine` 类是系统的“大脑”，负责接收外部指令并调度内部组件。它维护了一个 `std::unordered_map<StockID, OrderBook*>` 来支持**多标的（Multi-Stock）**交易。

*   **指令解析**: `processCommand()` 函数解析文本指令。
    *   **ADD** `stock_id order_id price qty side`: 创建订单并路由到指定 `StockID` 的 `OrderBook`。
    *   **CANCEL** `stock_id order_id`: 路由到指定 `StockID` 的 `OrderBook` 进行撤单。
    *   **SNAPSHOT**: 手动触发快照。
*   **自动快照**:
    *   每次处理订单后，`processedOrderCount` 计数器递增。
    *   当计数器达到 `SNAPSHOT_INTERVAL` 阈值时，自动调用 `triggerSnapshot()`。
*   **路由分发**:
    *   通过 `StockID` 在 HashMap 中查找对应的 `OrderBook`。
    *   如果不存在，则**惰性创建**一个新的 `OrderBook` 实例。
*   **快照触发器 (`triggerSnapshot`)**:
    *   核心逻辑：调用 `fork()`。
    *   **父进程**: 立即返回，继续处理下一条指令（非阻塞）。
    *   **子进程**: 遍历所有活跃的 `OrderBook`，分别调用 `OrderBook::saveSnapshot` 将其状态保存为独立文件（如 `snapshot_stock_1_pid.dat`），完成后直接退出。

### Step 3: 订单簿撮合 (`src/engine/OrderBook.cpp`)

`OrderBook` 是系统的“心脏”，维护单只股票的买卖队列并执行撮合。

1.  **新订单进入 (`addOrder`)**:
    *   **内存分配**: 从 `ObjectPool` 中申请一个 `Order` 对象（避免 `new` 带来的系统开销）。
    *   **尝试撮合 (`matchOrder`)**:
        *   买单 vs 卖单队列 (`asks`)，卖单 vs 买单队列 (`bids`)。
        *   如果不满足价格条件（未穿价），停止撮合。
        *   如果成交，更新双方剩余数量 (`leavesQty`)。
    *   **挂单 (`Insert`)**:
        *   如果订单未完全成交，将其插入到 `SkipList`（跳表）中对应的 `PriceLevel`（价格档位）。
2.  **撤单 (`cancelOrder`)**:
    *   通过 `HashMap` (`std::unordered_map`) 进行 $O(1)$ 查找。
    *   从 `PriceLevel` 的链表中 $O(1)$ 移除。
    *   将对象归还给 `ObjectPool`。

## 3. 关键文件职责详解

### 3.1 基础组件 (`include/common/`)

*   **`Types.h`**:
    *   **作用**: 定义了系统的基本类型（固定宽度的 `int64_t` 价格、`uint64_t` 订单ID等），确保跨平台一致性。
    *   **核心**: `enum class Side` (买売方向), `enum class OrderType` (限价/市价)。

*   **`Order.h`**:
    *   **作用**: 定义订单的数据结构。
    *   **核心特性**: **侵入式链表节点** (`prev`, `next` 指针)，使得订单对象本身就是链表节点，无需额外的容器节点内存分配，极大提升缓存命中率。

*   **`ObjectPool.h`**:
    *   **作用**: 高性能内存池。
    *   **逻辑**: 预先申请大块内存（Block），通过空闲链表（FreeList）重复利用已销毁的对象，避免高频 `malloc/free` 造成的碎片和延迟。

*   **`Logger.h`**:
    *   **作用**: 简单的日志工具，提供带时间戳的标准输出。

### 3.2 引擎核心 (`include/engine/` & `src/engine/`)

*   **`Engine.h/cpp`**:
    *   **作用**: 应用程序框架，负责 I/O 交互、配置管理和进程管理（Fork）。
    *   **核心**:
        *   `std::unordered_map<StockID, OrderBook*>`: 管理多只股票。
        *   `fork()`: 实现了“快照不卡顿”的关键需求。
        *   **配置加载**: 解析 `.conf` 文件，减少硬编码。

*   **`OrderBook.h/cpp`**:
    *   **作用**: 业务逻辑核心。
    *   **数据结构组合拳**:
        *   **HashMap**: 用于 $O(1)$ ID 查询。
        *   **SkipList**: 用于维持价格优先级的有序队列（插入/查找 $O(\log N)$）。
        *   **PriceLevel**: 具体的每个价格档位，内部是 FIFO 队列。
    *   **快照逻辑**: 负责将当前内存状态序列化为二进制文件。

*   **`SkipList.h`**:
    *   **作用**: 手写实现的模版跳表。
    *   **逻辑**: 替代红黑树（`std::map`）作为有序容器。相比红黑树，跳表在并发场景（虽然本项目是单线程）和实现复杂度上更优，且更可控。
    *   **比较器**: `template <typename Comparator>` 支持买单从高到低排序，卖单从低到高排序。
