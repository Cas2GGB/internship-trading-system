# 实习课题：交易系统快速内存快照方案调研与原型实现

**指导人**：开发组组长
**实习周期**：3周
**课题性质**：系统编程 / 性能优化

---

## 1. 课题背景
在交易撮合系统中，当系统出现故障或需要进行历史回溯时，我们需要快速获取当前进程的内存状态（快照）。
本课题的目标是寻找一种既能保存完整内存数据，又不会导致交易主线程长时间卡顿的方案。我们将重点对比“外部系统工具”与“内部代码优化”两种实现路径的性能差异。

## 2. 核心目标
1.  **调研**：分析 Linux 下常见的内存导出方式（如 `gcore`、`/proc` 文件系统、`fork` 机制）的优缺点。
2.  **原型**：开发一个单线程的简易撮合引擎，并实现基于 `fork`（写时复制）的高效快照功能。
3.  **对比**：量化分析“无快照”、“外部工具快照”、“Fork 快照”三种模式对交易处理性能的具体影响。

---

## 3. 详细计划安排

### 第一周：技术调研与方案验证
**本周重点**：理解问题，熟悉 Linux 内存机制，并通过手动实验验证“为什么直接拷贝内存行不通”。

### 第二周：撮合引擎原型与快照开发
**本周重点**：编写核心代码，实现一个具备“自我快照”能力的简易交易系统。

### 第三周：性能压测与数据分析
**本周重点**：通过数据证明 `fork` 方案的优势，完成结题报告。

---

## 4. 实习要求
1.  **开发环境**：Linux 系统。
2.  **数据一致性**：快照导出的文件必须能被重新加载，且恢复后的账户金额/持仓数据必须与快照时刻完全一致。
3.  **简化原则**：不必追求复杂的序列化协议，能把数据存下来并读回去即可；重点在于流程跑通和性能数据的获取。




----
* **Linux 内存机制学习**：
    * 学习 Linux 的虚拟内存管理、页表（Page Table）以及 `/proc/[pid]/maps` 和 `/proc/[pid]/mem` 的作用。
    * 理解 `fork` 系统调用的 Copy-On-Write（写时复制）原理。
* **外部工具实验**：
    * 编写一个简单的“内存占用模拟程序”（不断申请内存并写入数据，模拟 1GB~2GB 占用）。
    * 手动使用 Linux 命令 `gcore [pid]` 对运行中的程序进行内存导出。
    * **关键任务**：直观感受并记录使用 `gcore` 时，目标进程被阻塞（卡死）了多久。
* **产出物**：
    * 一份简短的《内存快照技术调研笔记》，阐述 `gcore` 等工具在低延迟场景下的局限性。


* **搭建简易撮合引擎**：
    * 使用 C++ (或 Rust/Go) 实现一个单线程程序。
    * 构建简单的订单簿（OrderBook）数据结构，支持处理下单指令。
    * 实现从文件读取历史指令并顺序执行的功能（作为后续性能测试的基准）。
* **实现 Fork 异步快照**：
    * 在代码中集成 `fork()` 调用。
    * 实现核心逻辑：
        * **父进程**：`fork` 后立即返回，继续处理下一笔交易，确保业务不中断。
        * **子进程**：遍历内存中的订单簿，将数据序列化写入磁盘文件，完成后退出。
* **产出物**：
    * 包含快照功能的源代码（可编译运行）。


* **性能测试环境搭建**：
    * 准备测试数据：生成 100 万条以上的模拟订单指令。
    * 设定测试场景：模拟 1GB 以上的大内存运行环境。
* **多组对比测试**：
    * **基准组**：不开启任何快照，跑完所有订单的耗时。
    * **对照组**：模拟使用外部脚本（如 gcore）导出内存，对交易处理造成的延迟影响。
    * **实验组**：开启 `fork` 异步快照，观察对交易吞吐量（TPS）的影响，以及快照触发瞬间的延迟抖动。
* **产出物**：
    * 《交易系统内存快照方案性能测试报告》（含对比图表和结论）。
    * 现场演示：在系统高速运行时触发快照，证明主流程未受明显阻碍。
