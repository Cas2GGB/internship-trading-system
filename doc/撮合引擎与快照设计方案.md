## 1. 概述

本方案旨在设计一个**轻量级、高吞吐的单线程撮合引擎**，并集成基于 Linux `fork()` COW (写时复制) 机制的**异步内存快照**功能。

系统核心目标：
1.  **高性能撮合**：利用 **跳表 (SkipList)** 和 **哈希表 (HashMap)** 实现订单的高效管理与查询。
2.  **低延迟快照**：利用操作系统 `fork` 机制实现毫秒级阻塞的内存快照，支持系统崩溃后的状态恢复。
3.  **模拟真实场景**：通过文件输入模拟高频交易指令，构建复杂的内存对象网络，以验证 `fork` 快照方案在较大内存占用下的性能表现。

## 3. 核心数据结构设计

为了模拟真实的高频交易系统并验证内存快照性能，我们将放弃简单的 `std::vector`，转而构建具有复杂指针引用的数据结构。

### 3.1 基础定义

```cpp
using OrderID = uint64_t;      // 订单ID，全局唯一，递增
using ClientID = uint64_t;     // 客户ID，用于区分不同用户的委托
using StockID = uint32_t;      // 股票代码，用于索引不同的订单簿
using Price = int64_t;         // 价格，放大10000倍的定点数 (20.5 -> 205000)
using Qty = uint32_t;          // 数量，通常为正整数

// 使用 uint8_t 节省内存空间
enum class Side : uint8_t { 
    BUY = 1,  // 买单
    SELL = 2  // 卖单
};
enum class OrderType : uint8_t { 
    LIMIT = 1,  // 限价单：指定价格成交
    MARKET = 2, // 市价单：按当前最优价成交
    CANCEL = 3  // 撤单指令
};
enum class TimeInForce : uint8_t { 
    GTC = 0, // Good-Till-Cancel: 一直有效直到由于交易或取消
    IOC = 1, // Immediate-Or-Cancel: 立即成交否则取消剩余部分
    FOK = 2  // Fill-Or-Kill: 全部成交否则全部取消
}; 

// === 新增：账户与风控系统 ===
struct Account {
    ClientID id = 0;                                 // 投资者唯一标识符 (客户ID)
    Price balance = 100000000;                       // 账户可用资金余额，默认初始化为 1 亿
    Price frozenFunds = 0;                           // 冻结资金 (当买单挂单未成交时，对应的预估金额被冻结)
    std::unordered_map<StockID, Qty> positions;      // 可用股票持仓
    std::unordered_map<StockID, Qty> frozenPositions;// 冻结的股票持仓
};
```

### 3.2 订单对象 (Order)

订单是系统中的最小单元。为了追求 Cache Line 友好，我们将字段按大小降序排列以减少内存 Padding。

```cpp
struct Order {
    // --- 8 Bytes aligned ---
    OrderID id;            // 订单唯一标识
    ClientID clientId;     // 发起该订单的客户ID
    Price price;           // 委托价格
    uint64_t timestamp;    // 委托时间戳 (纳秒)
    
    // --- 4 Bytes aligned ---
    Qty originalQty;       // 原始委托数量
    Qty leavesQty;         // 剩余未成交数量 (初始值等于 originalQty，随成交减少)
    StockID stockId;       // 该订单所属的股票代码

    // --- 1 Byte aligned ---
    Side side;             // 买卖方向
    OrderType type;        // 订单类型
    TimeInForce timeInForce; // 执行策略 (GTC/IOC/FOK)
    
    // --- Pointers (8 Bytes on 64-bit OS) ---
    // 侵入式链表指针：用于在 PriceLevel 中串联同价格订单
    // 这种设计允许我们在 O(1) 时间内从链表中移除特定订单
    Order* prev;           // 指向链表中的前一个订单
    Order* next;           // 指向链表中的后一个订单

    // 反向指针：指向所属的价格档位，方便撤单时 O(1) 更新档位统计信息
    struct PriceLevel* level; 
};
```

### 3.3 价格档位 (PriceLevel)

将相同价格的订单组织在一起，形成一个队列，遵循**时间优先**（FIFO）原则。

```cpp
struct PriceLevel {
    // ... (现有字段) ...
    
    // 【时间优先体现】
    // 同一价格档位内，订单通过双向链表维护先后顺序。
    // 新订单：总是插入到 tail 后 (Push Back)。
    // 撮合成交：总是从 head 开始吃单 (Pop Front)。
    Order* head;       // 链表头（最早的订单，最先成交）
    Order* tail;       // 链表尾（最新的订单，最后成交）
    
    // ...
};
```

### 3.4 订单簿 (OrderBook)

一个 `OrderBook` 实例对应一个交易标的/股票代码 (`StockID`)。

```cpp
class OrderBook {
    // ...
    
    // 【价格优先体现】
    // 核心存储：Bid/Ask 跳表
    // SkipList 是一种有序数据结构，其遍历顺序即为撮合顺序。
    
    // 买单簿 (Bids)：使用 Greater 比较器，价格从 高 -> 低 排序。
    // 撮合时：从头节点开始遍历，即优先撮合出价最高的买单。
    SkipList<PriceLevel, Greater> bids;  
    
    // 卖单簿 (Ask)：使用 Less 比较器，价格从 低 -> 高 排序。
    // 撮合时：从头节点开始遍历，即优先撮合要价最低的卖单。
    SkipList<PriceLevel, Less> asks;     
    
    // ...
};
```

### 3.5 核心容器选型

系统同时维护两套索引，以支持不同的业务操作：

#### A. 价格优先索引：跳表 (Skip List)
*   **用途**：维护买卖盘的深度数据 (`OrderBook`)。
*   **结构**：
    *   **Ask Book (卖方)**：按价格**从小到大**排序的跳表。
    *   **Bid Book (买方)**：按价格**从大到小**排序的跳表。
*   **选型理由**：
    1.  **查询效率**：平均时间复杂度 `O(log N)`，与红黑树相当。
    2.  **实现简便**：相比红黑树复杂的旋转操作，跳表更容易实现，且更易于进行内存布局优化（对于快照来说，连续的内存更友好，但也支持这种指针链接的结构）。
    3.  **遍历高效**：跳表的第0层就是一个有序链表，非常适合撮合引擎按“价格优先”逐个档位吃单。
*   **操作复杂度**：查找最优价 `O(1)`，插入新档位 `O(log N)`。

#### B. 订单索引：哈希表 (Hash Map)
*   **用途**：支持根据 `OrderID` 快速查找订单。
*   **结构**：`std::unordered_map<OrderID, Order*>`
*   **选型理由**：我们需要支持**快速撤单 (Cancel)**。
    *   如果没有 Hash Map，撤单需要遍历 OrderBook，复杂度 `O(N)`。
    *   使用 Hash Map，可以直接定位到 `Order*` 指针，结合双向链表结构，实现 `O(1)` 复杂度的撤单。

### 3.6 内存管理：对象池 (Object Pool)

为了实现**零 GC (Zero Garbage Collection)** 和 **极致低延迟**，系统**严禁在交易主循环中使用 `new`/`delete`**。

#### A. 设计目标
1.  **消除系统调用**：避免 `malloc/free` 带来的不确定性延迟。
2.  **缓存亲和性 (Cache Locality)**：让 Order 对象在物理内存上连续存放，提升 L1/L2 Cache 命中率。
3.  **大页支持 (Huge Pages)**：配合 Linux HugePages 使用，减少 TLB Miss。

#### B. 实现方案
采用 **Freelist (空闲链表)** 机制管理预分配内存块。

```cpp
template <typename T, size_t PoolSize>
class ObjectPool {
private:
    std::vector<T> data;      // 预分配的连续内存块 (Pool)
    std::vector<T*> freeList; // 空闲对象指针栈

public:
    ObjectPool() {
        data.reserve(PoolSize); // 确保 vector 不会扩容重分配
        for (size_t i = 0; i < PoolSize; ++i) {
            data.push_back(T());
            freeList.push_back(&data.back());
        }
    }

    // O(1) 分配
    T* allocate() {
        if (freeList.empty()) return nullptr; // 或扩容策略
        T* ptr = freeList.back();
        freeList.pop_back();
        return ptr;
    }

    // O(1) 回收
    void deallocate(T* ptr) {
        freeList.push_back(ptr);
    }
};

// 全局实例
ObjectPool<Order, 1000000> orderPool;       // 预分配 100万个订单对象
ObjectPool<PriceLevel, 50000> levelPool;    // 预分配 5万个价格档位
```

#### C. 与 Fork 快照的协同
*   使用对象池后，`fork` 时的内存页表映射更加规整。
*   由于所有 `Order` 都集中在几个大的 Virtual Memory Area (VMA) 中，`fork` 复制页表的效率相比散乱的 `new` 分配会有显著提升。

## 4. 关键业务流程

### 4.1 限价单处理 (Limit Order Processing)

1.  **查验 (Risk Control - 风控检查)**：
    *   检查订单参数合法性（比如客户是否存在，预先获取 `Account` 指针）。
    *   **买单 (Buy)**：校验 `Account.balance >= (price * qty)`，若不足则拒单；校验通过后扣减可用 `balance` 并增加 `frozenFunds`。
    *   **卖单 (Sell)**：校验 `Account.positions[StockID] >= qty`，若不足则拒单；校验通过后扣减可用 `positions` 并增加 `frozenPositions`。
    *   *注*：为了满足百万级压测，提供配置 `ENABLE_STRESS_TEST=1` 直接绕过上述资金逻辑，以避免大量单向测试用例被全部拒绝而产生不准确的 TPS 指标。
2.  **尝试撮合 (Match)**：
    *   根据订单方向，获取对手方 OrderBook 的**最优价节点**（跳表头节点）。
    *   **循环判断**：若对手价满足成交条件（买单价 >= 卖一价 或  卖单价 <= 买一价）：
        *   生成的成交记录 (Trade)。
        *   修改内存：扣减对手单 `leavesQty`，更新 `PriceLevel.totalQty`。
        *   **清理**：若对手单完全成交，利用 `prev/next` 指针将其从链表移除，并从 Hash Map 中删除；若该 `PriceLevel` 为空，从跳表中删除该节点。
        *   扣减本单 `leavesQty`。
        *   若本单全部成交，结束流程。
3.  **挂单 (Post)**：
    *   若还有剩余 `leavesQty`，需要在本方 OrderBook 中挂单。
    *   **查找/插入档位**：在跳表中查找该 Price 的节点。如果不存，则创建一个新的 `PriceLevel` 并插入跳表。
    *   **插入链表**：将 Order 追加到 `PriceLevel` 的链表尾部。
    *   **建立索引**：将 `OrderID -> Order*` 插入 Hash Map。

### 4.2 撤单处理 (Cancel Order Processing)

1.  **查找**：在 Hash Map 中查找 `OrderID`。若未找到，返回“订单不存在”。
2.  **定位**：获取 `Order*` 指针。
3.  **解绑**：
    *   利用 `Order.prev` 和 `Order.next` 将其从双向链表中摘除。
    *   利用 `Order.level` 指针找到所属 `PriceLevel`，扣减 `totalQty`。
4.  **清理**：
    *   若 `PriceLevel` 的订单数为0，将该 Level 从跳表中移除并释放内存。
    *   从 Hash Map 中移除该 ID。
    *   释放 Order 内存。
5.  **效率**：整个过程仅涉及哈希查找和指针操作，**时间复杂度严格为 O(1)**。

## 5. 快照与恢复方案 (Fork + COW)

这是本课题的核心实验部分。本方案本质上是**全量快照**（子进程将完整状态写入磁盘），但利用操作系统的 **COW (Copy-On-Write)** 机制实现了**极低延迟**的创建过程。

### 5.1 核心原理：为什么选择 Fork？

很多初学者容易误解 `fork` 会导致性能问题，实际上在 Linux 下它是实现快照的神器：

1.  **逻辑上是“全量复制”**：
    *   子进程在逻辑上拥有和父进程一模一样的内存空间。
    *   子进程遍历内存写文件时，写的是 `fork` 那一瞬间的“冻结”状态。
2.  **物理上是“写时复制 (COW)”**：
    *   **创建瞬间 (Fork)**：操作系统**并不复制**物理内存数据，仅仅复制了**页表 (Page Table)**。
        *   页表是将虚拟地址映射到物理地址的目录。
        *   复制页表非常快（通常 < 1ms），因此主线程几乎不阻塞。
    *   **运行期间 (Run-time)**：
        *   Fork 后，父子进程指向**同一块物理内存**，且所有页面被 OS 标记为 **Read-Only**。
        *   **子进程**：只读不写（Dump数据），直接读取共享物理内存。
        *   **父进程**：继续处理交易。当它尝试**修改**某个订单（写内存）时，硬件触发 CPU 异常（Page Fault）。操作系统捕获异常，将**该特定页面 (4KB)** 复制一份新的给父进程写入。
    *   **结论**：父进程只为“被修改的数据”支付微小的复制成本，而未修改的海量历史数据（冷数据）始终共享，从而保证了交易系统的低延迟。

### 5.2 快照触发 (Snapshot Trigger)
*   **触发时机**：
    *   按时间间隔（如每分钟）。
    *   按处理条数（如每 100 万单）。
*   **执行动作**：调用 `fork()` 系统调用。

### 5.3 父子进程行为

*   **父进程 (Parent)**：
    *   `fork()` 返回子进程 PID。
    *   **立即返回**主循环，继续处理下一条指令。
    *   *性能损耗点*：仅在于 `fork` 复制页表的时间（通常 < 1ms），以及后续**写操作**时触发缺页中断（Page Fault）进行物理页复制的微小开销。

*   **子进程 (Child)**：
    *   `fork()` 返回 0。
    *   **静止的内存视图**：子进程拥有 fork 瞬间父进程完全一致的内存状态。
    *   **序列化任务**：
        1.  创建文件 `snapshot_<timestamp>.dat`。
        2.  写入 Header (Magic, Version, LastSequenceID)。
        3.  **遍历数据**：
            *   遍历 Ask SkipList (第0层链表)，将每个 PriceLevel下的每个 Order 写入文件。
            *   遍历 Bid SkipList (同上)。
            *   *注意*：**不通过 Hash Map 遍历**，以保证价格有序性，方便恢复。
            *   *序列化格式*：只存数据 (ID, Price, Qty, ...)，不存指针地址。
        4.  **退出**：任务完成后调用 `_exit(0)`，避免触发父进程注册的析构函数。

### 5.4 恢复流程 (Recovery)

1.  系统启动时读取快照文件。
2.  读取 Header，初始化 SequenceID。
3.  **内存重建**：
    *   逐条读取 Order 数据。
    *   **重建索引**：每读一条，调用内部的 `insert()` 函数，重新在堆内存中分配对象，插入新的 SkipList Node 和 Hash Map Entry。
    *   *关键点*：文件里没有指针，指针是在恢复过程中重新 `new` 出来的，因此内存地址会变，但逻辑拓扑结构与快照时一致。
4.  快照加载完毕后，打开指令日志文件，**Seek** 到 `LastSequenceID` 之后的位置继续回放。

## 7. 日志与监控系统设计

为满足“真实还原交易现场”的需求，设计一套结构化的文本日志系统。

### 7.1 日志格式
格式：`[TIMESTAMP] [LEVEL] [MODULE] EVENT | KEY=VALUE ...`

### 7.2 关键事件定义

1.  **订单接收 (OrderAccepted)**
    *   `[INFO] [ENG] ORDER_NEW | oid=1001 cid=888 inst=BTC-USDT side=BUY price=50000 qty=1`
2.  **成交 (TradeExecuted)**
    *   `[INFO] [MAT] TRADE_FILL | match_id=5001 maker_oid=998 taker_oid=1001 price=50000 qty=1`
3.  **撤单 (OrderCanceled)**
    *   `[INFO] [ENG] ORDER_CXLD | oid=998 reason=USER_REQUEST`
4.  **快照 (Snapshot)**
    *   `[INFO] [SNP] FORK_START | parent_pid=1234 seq_id=10500`
    *   `[INFO] [SNP] DUMP_DONE  | child_pid=1235 file=snap_10500.bin size_mb=256 time_ms=45`

### 7.3 实现策略
*   使用 `std::cerr` 或轻量级文件流。
*   **缓冲机制**：设置 4KB ~ 64KB 的应用层 Buffer，减少 syscall 调用频率，避免 I/O 阻塞交易主线程（模拟真实生产环境的 Async Logger 行为）。

## 9. 风险控制与实现细节⚠️

在工程实现阶段，需特别注意以下 C++ 系统编程陷阱：

1.  **Fork 死锁风险**：
    *   `fork()` 只复制调用线程，其他线程在子进程中瞬间消失。
    *   若父进程的日志库或 `malloc` 正持有锁，子进程继承了该锁状态但无法释放，导致死锁。
    *   **对策**：子进程中禁用 `LOG` 宏，仅使用 `write` 系统调用；尽量避免在子进程分配内存。

2.  **文件写入原子性**：
    *   避免快照文件只写了一半导致损坏。
    *   **对策**：先写入 `temp.snap`，由于 `rename` 是原子操作，写入完成后执行 `rename("temp.snap", "final.snap")`。

3.  **ID生成器持久化**：
    *   恢复后必须知道下一个可用的 `OrderID`。
    *   **对策**：快照 Header 必须显式记录 `NextOrderID`。

4.  **恢复逻辑隔离**：
    *   加载快照时，订单是“历史遗留挂单”，绝对**不能触发撮合**。
    *   **对策**：实现专用的 `loadOrder()` 接口，跳过 Matching 逻辑，只做 Insert。

## 10. 行业对比分析：本方案 vs 商业级引擎

为了让实习报告更具深度，我们将本原型与真实生产环境的撮合引擎进行对比：

| 特性 | 本原型方案 (Prototype) | 商业级撮合引擎 (Production) | 差异原因 |
| :--- | :--- | :--- | :--- |
| **网络模型** | **文件输入 (File I/O)** | **UDP组播 / TCP网关** | 剥离网络干扰，专注于内存本身 |
| **持久化** | **Fork 快照 (COW)** | **异步命令日志 (Command Log) + 周期快照** | 商业系统不仅要快照，还要每一笔都落盘 (WAL) 以防断电丢数据 |
| **核心数据结构** | **跳表 (SkipList)** | **红黑树 (RB-Tree) / 数组二分** | 各家流派不同，但目标都是 $O(1)/O(\log N)$ |
| **内存管理** | **定长对象池** | **Hierarchical Memory Pool** | 生产环境需处理不定长数据 (如变长字符串) |
| **并发模型** | **单线程 (Single Thread)** | **单线程 + RingBuffer 流水线** | 核心撮合逻辑完全一致 (一定是单线程)，周围配套设施不同 |
| **容灾恢复** | **加载快照 (<1s)** | **双机热备 (Active-Standby)** | 商业系统通常有备机实时同步内存状态 |

1.  **Day 1: 基础设施搭建**
    *   实现 `SkipList` 模板类 (主要支持 Insert, Delete, Search)。
    *   定义 `Order`, `PriceLevel` 结构。
2.  **Day 2: 撮合引擎核心**
    *   实现 `Engine::placeOrder` (Match + Add)。
    *   实现 `Engine::cancelOrder` (Hash lookup + Unlink)。
3.  **Day 3: 业务流程验证**
    *   编写简易的数据生成器（生成 10 万条下单/撤单指令）。
    *   运行 Engine，验证撮合逻辑正确性，确保无内存泄漏。
4.  **Day 4: 快照功能开发**
    *   实现 `SnapshotManager::takeSnapshot` (Fork + File Write)。
    *   实现 `SnapshotManager::loadSnapshot` (File Read + Rebuild)。
5.  **Day 5: 综合测试**
    *   流程：运行 -> 快照 -> 停止 -> 加载快照 -> 继续运行。
    *   验证：恢复后的内存状态（跳表结构、订单数）是否与快照时完全一致。
